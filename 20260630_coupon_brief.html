ทำคูปอง & โปรโมชันให้ดีขึ้น

เราจะทำให้ระบบส่วนลดดีขึ้นยังไง โดยปรับให้เข้ากับของที่เรามีจริง — เอกสารนี้สรุปทางเลือก, ข้อแลกเปลี่ยน, ข้อเสนอแนะ และเรื่องที่ต้องตัดสินใจ. อ่านได้โดยไม่ต้องเป็นสายเทคนิค — ศัพท์ยาก ๆ มีอธิบายในหัวข้อ "ศัพท์เทคนิค".

สถานะ: กำลังดำเนินการ ขั้น: Discovery (ศึกษา/ตัดสินใจ) เจ้าภาพ: Backend / Product ความเสี่ยง: ปานกลาง อัปเดต: 2026-06-30
สรุป (อ่านก่อน) ศัพท์เทคนิค วันนี้เรามีอะไร 3 ทางเลือก ตารางเทียบทางเลือก ข้อเสนอแนะ + เหตุผล แผนทำเป็นก้อน ความเสี่ยง + วิธีคุม สิ่งที่ต้องตัดสินใจ
1ประเภทส่วนลดวันนี้ลดเป็นบาทคงที่อย่างเดียว
3ทางเลือกที่ประเมินต่อยอด / เพิ่มชั้น / รื้อใหม่
Aทางเลือกที่แนะนำต่อยอดของเดิมทีละนิด

สรุปสั้น ๆ

  • ของเดิมใช้งานได้จริงแล้ว: admin สร้าง/ลบคูปอง, เจ้าหน้าที่กรอกโค้ดตอนสร้าง booking แล้วระบบลดราคา-บันทึกบัญชีถูกต้อง. แต่ตั้งใจทำ ฟีเจอร์น้อย: ลดได้แค่เป็นบาทคงที่, เราออกเงินส่วนลดเสมอ, แก้คูปองไม่ได้, ไม่มีคูปองผูกลูกค้าเฉพาะคน.
  • ดีไซน์อ้างอิง (doc.md) เสนอระบบใหญ่เต็มรูป — แต่หลายส่วน ยังไม่มีที่ใช้ เพราะเราไม่มีแอป/หน้าจ่ายเงินฝั่งลูกค้า (เช่น กระเป๋าคูปอง, เลือกคูปองอัตโนมัติ, จับโกง).
  • แนะนำทางเลือก A — ต่อยอดของเดิมทีละนิด (ทำน้อย-เสี่ยงน้อย-ได้เร็ว). ปัญหาทุกข้อเป็นแค่ "เติมฟีเจอร์" ไม่ใช่ "ออกแบบผิด".
  • ทำเป็นก้อน ๆ: ก้อน 1 ลด % → ก้อน 2 แก้/พักคูปอง + เก็บประวัติ → ก้อน 3 voucher ชดเชยลูกค้า → ก้อน 4 ให้เจ้าของที่พักออกส่วนลด (รอการเงิน) → ก้อน 5 เพดานงบ (ทำหรือไม่ก็ได้).
  • เรื่องที่ต้องตัดสินก่อน: มีแผนทำแอปฝั่งลูกค้าไหม (กำหนดว่าหลายฟีเจอร์มีที่ใช้หรือไม่) + เจ้าของที่พักออกส่วนลดแล้วหักเงินจากใคร.
ศัพท์เทคนิค — อธิบายแบบเข้าใจง่าย
คูปอง (coupon/promotion)โค้ดส่วนลดที่ admin สร้างให้ใช้ตอนจอง เช่น SUMMER1000 ลด 1,000 บาท
voucherคูปองที่ผูกกับลูกค้าเฉพาะคน ไม่ใช่โค้ดสาธารณะ — เช่นคูปองชดเชยที่ support ออกให้ลูกค้ารายนั้นโดยเฉพาะ
ลดเป็นบาทคงที่ (flat)ลดเป็นจำนวนเงินตายตัว เช่น "ลด 500 บาท"
ลดเป็น % + เพดานลดลดเป็นเปอร์เซ็นต์แต่ไม่เกินเพดาน เช่น "ลด 10% สูงสุด 500 บาท"
ยอดขั้นต่ำ (min spend)ต้องจองถึงยอดนี้คูปองถึงใช้ได้ เช่น "จองครบ 3,000"
ใครออกเงินส่วนลด (funded by)ส่วนลดนี้ใครเสียเงิน — แพลตฟอร์ม (เรา) หรือเจ้าของที่พัก
เงินที่ต้องจ่ายเจ้าของที่พักยอดที่บริษัทต้องโอนคืนเจ้าของที่พักหลังหักค่าธรรมเนียม. ถ้าเจ้าของออกส่วนลดเอง ส่วนลดจะไปหักตรงนี้
การใช้คูปอง (redemption)บันทึก 1 รายการว่า "คูปองนี้ถูกใช้กับ booking ไหน ลดเท่าไหร่" — ตรวจย้อนหลังได้
บัญชีเงิน — เขียนเพิ่มอย่างเดียว (ledger)สมุดบันทึกการเคลื่อนไหวเงินของ booking — เขียนเพิ่มได้อย่างเดียว ห้ามลบ/แก้ย้อนหลัง เหมือนสมุดบัญชีจริงที่ขีดฆ่าไม่ได้
ทำเป็นชุดเดียว (transaction)งานหลายขั้น (สร้าง booking + ลดราคา + บันทึกการใช้คูปอง) สำเร็จพร้อมกันหรือยกเลิกทั้งหมด — ไม่มีค้างครึ่งทางที่เงินเพี้ยน
ตัวกันคนกดชนกัน (lock)กันสองคนกดใช้คูปองเดียวกันพร้อมกันแล้วเกินโควต้า — ระบบล็อกให้ทำทีละคน
การปรับฐานข้อมูล (migration)การเพิ่ม/แก้ช่องเก็บข้อมูล. แบบ "เพิ่มช่องใหม่เฉย ๆ ไม่แตะของเดิม" = เสี่ยงต่ำ
จุดเชื่อมระบบ (port/contract)จุดต่อระหว่างระบบจองกับระบบคูปอง — ถ้าไม่แตะจุดนี้ ฝั่งจองไม่ต้องแก้ตาม
ตรวจก่อนแล้วค่อยใช้เจ้าหน้าที่พิมพ์โค้ด → กดตรวจ → ระบบบอกว่าใช้ได้/ไม่ได้และลดเท่าไหร่ → ค่อยกดใช้จริง
กระทบยอด (reconcile)ฝ่ายการเงินตรวจว่าตัวเลขเงินทุกฝั่งตรงกัน — ส่วนลดต้องบันทึกถูกไม่งั้นกระทบยอดไม่ลง
ระบบก้อนเดียวแบ่งโมดูล + ADR-001สถาปัตยกรรมปัจจุบัน: ระบบเดียว ฐานข้อมูลเดียว แบ่งโมดูลในตัว — เลือกไว้โดยตั้งใจ. ADR-001 = กติกาบริษัทว่า "ห้ามสร้างใหญ่เกินก่อนมีความต้องการจริง"
วันนี้เรามีอะไร (และยังขาดอะไร)

✅ ทำได้แล้ว

  • admin สร้าง + ลบคูปองได้
  • เจ้าหน้าที่กรอกโค้ดตอนสร้าง booking → ตรวจ → ลดราคา
  • ส่วนลดบันทึกลงบัญชีเงินถูกต้อง (เป็นชุดเดียว, มีตัวกันคนกดชนกัน, กันคูปองซ้ำต่อ booking)
  • เวลาคูปองใช้ไม่ได้ ระบบบอกเหตุผลชัด (หมดอายุ/ยอดไม่ถึง/ฯลฯ)

❌ ยังขาด

  • ลดเป็น % (ทำได้แค่เป็นบาทคงที่)
  • แก้คูปองหลังสร้าง (มีแค่สร้าง + ลบ)
  • ให้เจ้าของที่พักออกส่วนลดเอง (เราออกเงินเสมอ)
  • voucher ผูกลูกค้าเฉพาะคน (สำหรับ support ชดเชย)
  • เพดานงบเป็นบาทต่อแคมเปญ
ข้อสรุปสำคัญ: ระบบเดิม "ทำงานถูกต้อง" — ที่ขาดคือ "จำนวนฟีเจอร์" ไม่ใช่ "ออกแบบผิด". จึงต่อยอดได้ ไม่ต้องรื้อ.
3 ทางเลือก

ทางเลือก A — ต่อยอดของเดิมทีละนิด แนะนำ

เก็บระบบเดิมไว้ แล้วเติมฟีเจอร์ที่ขาดทีละอย่าง โดยไม่รื้ออะไร — "ทำน้อย เสี่ยงน้อย ได้เร็ว".

PO ได้ฟีเจอร์ที่อยากได้เร็ว ทีละก้อน ปล่อยใช้ได้เรื่อย ๆ; ความเสี่ยงเงินเพี้ยนต่ำสุด; ตรงกติกาบริษัท
PO เสียยังทำ "1 แคมเปญหลายโค้ด" ไม่ได้ — ถ้าวันหน้าจำเป็น ต้องกลับมาปรับโครงอีกรอบ
เวลา: ~1 สัปดาห์/ก้อน · ความเสี่ยง: ปานกลาง (คุมได้ด้วยการทดสอบ)

ทางเลือก B — เพิ่มชั้นใหม่ครึ่งทาง ทำกลาง

เก็บคูปองเดิมเป็นแม่แบบกฎ แล้วเพิ่มชั้นใหม่เพื่อรองรับ "1 แคมเปญ → หลายโค้ด" และ voucher ผูกลูกค้าเต็มตัว.

PO ได้ความยืดหยุ่นแบบที่ทางเลือก A ทำไม่ได้ (หลายโค้ดต่อแคมเปญ + voucher ลูกค้าแบบเต็มตัว)
PO เสียต้องไปแก้จุดที่ต่อกับระบบเงินที่ทำงานถูกอยู่ = เสี่ยงพังของเดิม; ช้ากว่า; กระเป๋าคูปองลูกค้ายังใช้ไม่เต็ม (ไม่มีแอป)
เวลา: สูง (1–2 sprint) · ความเสี่ยง: สูง (แตะเส้นทางเงิน)

ทางเลือก C — รื้อสร้างใหม่ทั้งหมดตาม doc.md ทำเยอะ

สร้างระบบใหม่ทั้งก้อน (แคมเปญ + คูปอง + voucher + งบ + ขออนุมัติ + จับโกง) ตามดีไซน์อ้างอิงเต็มรูป.

PO ได้ครบทุกอย่างใน doc.md
PO เสียเกือบทั้งหมดยังไม่มีคนใช้ (ไม่มีแอปลูกค้า); ช้าหลายเดือนกว่าจะได้ของ; ต้องรื้อระบบเงินที่ทำงานถูก = เสี่ยงสุด; ขัดกติกาบริษัท
เวลา: สูงมาก (หลาย sprint หลายทีม) · ความเสี่ยง: สูงมาก
ตารางเทียบทางเลือก (ภาษาคน)
เกณฑ์A — ต่อยอดB — เพิ่มชั้นC — รื้อใหม่
ได้ของชิ้นแรกเร็วแค่ไหนเร็ว (~1 สัปดาห์/ก้อน)ช้าช้ามาก
ตรงกติกาบริษัท (ไม่สร้างใหญ่เกิน)ตรงก้ำกึ่งขัด
โอกาสทำของเดิมพังต่ำสูงสูงมาก
ต้องแก้ฝั่งระบบจองไหมไม่ต้องต้องแก้รื้อใหม่
รองรับ "1 แคมเปญหลายโค้ด"ไม่ได้ได้ได้
กระเป๋าคูปองลูกค้า (เมื่อมีแอป)ต้องเพิ่มหน้าจอมีพร้อมมีครบ
ถ้าแอปลูกค้าไม่มา จะเหลือภาระแค่ไหนต่ำกลางสูง (ครึ่งระบบไม่ถูกใช้)
ปรับทิศทางทีหลังง่ายแค่ไหนง่ายกลางยาก
ข้อเสนอแนะ + เหตุผล

เลือกทางเลือก A — ต่อยอดของเดิมทีละนิด

  • ของเดิมออกแบบมาดีแล้ว ที่ขาดคือจำนวนฟีเจอร์ — ลดราคา + ลงบัญชีเงิน + กันข้อมูลปน + กันคนกดชนกัน ทำถูกอยู่แล้ว.
  • ช่อง "ประเภทส่วนลด" และ "ใครออกเงิน" เผื่อไว้ในดีไซน์แล้ว → เพิ่มลด % และให้เจ้าของที่พักออกส่วนลด คือการใช้ของที่มี ไม่ใช่สร้างใหม่.
  • กติกาบริษัทห้ามสร้างใหญ่เกินก่อนมีความต้องการจริง — ทางเลือก C สร้างระบบเยอะทั้งที่ยังไม่มีสัญญาณ (ไม่มีทีมการตลาดเต็มตัว/ไม่มีแอปลูกค้า/ยอดยังไม่เยอะ).
  • เรายังไม่มีแอปฝั่งลูกค้า → ครึ่งหนึ่งของ doc.md ยังไม่มีที่ใช้ สร้างตอนนี้ = ของที่ไม่มีใครใช้.
  • ปรับทิศทีหลังได้ถูก — ออกแบบเผื่อไว้ ถ้าวันหน้าต้องการมากขึ้น ค่อยขยับไปทางเลือก B บนความต้องการจริง.
แผนทำเป็นก้อน (ของทางเลือก A)

  1. ลดเป็น % + เพดานลด

    เช่น "ลด 10% สูงสุด 500" — ผลกระทบสูงสุด ทำง่ายสุด

    1–3 วัน

  2. แก้/พักคูปอง + เก็บประวัติ

    ปิดปัญหา "แก้คูปองไม่ได้"; รู้ว่าใครแก้ตอนไหน

    2–4 วัน

  3. voucher ชดเชยลูกค้า

    support ออก voucher ผูกลูกค้า + เหตุผล + มีประวัติ

    3–5 วัน

  4. เจ้าของที่พักออกส่วนลด

    ต้องให้การเงินยืนยันวิธีหักเงินก่อน (CL-04)

    2–4 วัน · รอการเงิน

  5. เพดานงบต่อแคมเปญ

    หยุดใช้เมื่อยอดส่วนลดรวมถึงเพดาน

    1–2 วัน · ทำหรือไม่ก็ได้
หมายเหตุ: ลำดับนี้เป็นข้อเสนอ — ลำดับจริงขึ้นกับว่า PO อยากแก้ปัญหาไหนก่อน (ดู "สิ่งที่ต้องตัดสินใจ").
ความเสี่ยง + วิธีคุม
ความเสี่ยงระดับวิธีคุม
ลด % แล้วการปัดเศษสตางค์ไม่ตรงบัญชีสูงปัดเศษให้เรียบร้อยก่อนบันทึก + ทดสอบว่าตัวเลขเงินทุกฝั่งตรงกันเป๊ะ
เจ้าของที่พักออกส่วนลดแล้วลงบัญชีผิดสูงหยุดก้อนนี้ไว้จนกว่าฝ่ายการเงินยืนยันว่าหักเงินจากใคร (CL-04) + ทดสอบกระทบยอด
เพิ่มช่องข้อมูลไปกระทบตัวกันคูปองซ้ำ/ตัวกันคนกดชนกันเดิมกลางเพิ่มช่องใหม่แบบไม่บังคับ ไม่แตะของเดิม + ทดสอบว่าคูปองเดิมยังใช้ได้
ขอบเขตงานบานไปทางเลือก Cกลางคุมด้วยคำถาม "มีแอปลูกค้าไหม" + กติกาบริษัท; ฟีเจอร์ที่ต้องมีแอป = ยกออกจนกว่าจะมีจริง
สิ่งที่ต้องตัดสินใจ
  • สำคัญCL-01 · มีแผนทำแอป/หน้าจอที่ลูกค้าใช้เองไหม? — กำหนดว่ากระเป๋าคูปอง/เลือกอัตโนมัติ/จับโกง มีที่ใช้หรือยกออกถาวร เจ้าภาพ: Product
  • สำคัญCL-02 · เพิ่ม "ลดเป็น % + เพดานลด" เป็นก้อนแรกไหม? — ผลกระทบการตลาดสูงสุด ทำง่ายสุด เจ้าภาพ: Product
  • ปานกลางCL-03 · เพิ่ม "แก้/พักคูปอง + เก็บประวัติ" ไหม? — แบบทำน้อย (เปิด/พักพอ) หรือทำเยอะ (สถานะครบ + ขออนุมัติ) เจ้าภาพ: Product
  • สำคัญCL-04 · เจ้าของที่พักออกส่วนลด — หักเงินจากใคร? — เจ้าของรับภาระ หรือแบ่งจ่ายร่วม (ต้องให้การเงินยืนยัน) เจ้าภาพ: Finance / Backend
