ระบบค่าธรรมเนียมตามเงื่อนไขของบ้าน + แยกมัดจำเป็นค่าห้อง

หนึ่งระบบ extensible สำหรับ "เงื่อนไขที่มีค่าธรรมเนียม" ต่อบ้าน — สัตว์เลี้ยง, ผู้เข้าพักเสริม, เด็ก, ผู้มาเยือน, ค่าทำความสะอาด, ค่าล้างจาน ฯลฯ — เก็บราคารายบ้าน, ตอนจองกลายเป็นบิลย่อยอัตโนมัติ, แก้หลังจองได้. คู่กับ แยกมัดจำ ให้คิด 60% จากค่าห้องอย่างเดียว ส่วน fee ทั้งหมดไปอยู่ในยอดเรียกเก็บคงเหลือ จ่ายทีเดียวจบ. หลักการ: reuse billing engine ของ value-added service เดิม (ไม่สร้าง engine ที่สอง) + จัดการในหน้า "กฎ/เงื่อนไข". (ยอดคงเหลือ = บิลที่ยังไม่จ่าย · ไม่มีระบบแจ้งเตือนลูกค้า)

ค่าห้อง ฿17,000 + แขกเสริม ฿1,500 (2×750) + สัตว์ ฿800 + ทำความสะอาด ฿500 มัดจำ 60% × ค่าห้อง = ฿10,200 งานโดยรวม: L (กว้าง/retest เยอะ)
สรุป (อ่านก่อน) ศัพท์ (ฉบับย่อ) ค่าสัตว์/แขกเสริม → บริการเสริม วันนี้ (As-Is) แบบใหม่ (Option B) Database schema ต้องแก้เยอะไหม แผนทำเป็นเฟส ความเสี่ยง + วิธีคุม เทียบ As-Is vs B สิ่งที่ต้องตัดสินใจ
฿10,200มัดจำตอนจอง60% ของ "ค่าห้อง" เท่านั้น (ไม่รวม fee)
฿9,600ยอดเรียกเก็บคงเหลือ (จ่ายทีเดียว)ค่าห้องคงเหลือ + แขกเสริม + สัตว์ + ทำความสะอาด
∞ชนิด fee เพิ่มได้ไม่จำกัดเพิ่ม code ใหม่ใน catalog · ไม่ต้องแก้โค้ด

อ่านสั้น ๆ ก่อน

  • concept กลาง = "รายการค่าธรรมเนียมของบ้าน" (Property Charge Item) — ไม่ใช่ "service": สัตว์เลี้ยง, ผู้เข้าพักเสริม, เด็ก, ผู้มาเยือน, ค่าทำความสะอาด, ค่าล้างจาน ฯลฯ = charge item ที่ตั้งราคารายบ้าน + มี category ที่กำหนด behavior ชัด. "บริการเสริมสมัครใจ" (อาหารเช้า/รถรับส่ง) เป็นแค่ หนึ่ง category ไม่ใช่ถังรวมทุกอย่าง.
  • หลักการ #1 — มี billing engine เดียว: ทุกอย่างที่แตะเงินไหลผ่าน flow เดียว (snapshot ราคา → charge → recompute) ที่ reuse จาก value-added engine. terms ห้ามมี logic คิดเงินของตัวเอง.
  • terms = ชั้น policy/อธิบาย ไม่ใช่ที่เก็บเงิน: "อนุญาตสัตว์/เด็ก/ผู้มาเยือนไหม, ห้ามสูบบุหรี่, quiet hours" อยู่ที่ property_terms. ถ้ากฎมีค่าใช้จ่าย → terms อ้างอิงไปยัง charge item (ไม่เก็บ fee_amount เดี่ยว ๆ) → เงินมีแหล่งเดียว.
  • แยก 2 หน้าที่ของ extra-guest/เด็ก: capacity gate (จองได้ไหม — base/max guest, เด็กนับ capacity ไหม) อยู่ที่ package; fee calc (ถ้าจองได้คิดเท่าไหร่) อยู่ที่ charge item. ไม่ยัดรวมที่เดียว.
  • แยกมัดจำ: มัดจำ = 60% ของค่าห้องอย่างเดียว · fee ทุกตัวเป็นบิลแยก (type fee) ไปอยู่ในยอดคงเหลือ จ่ายปุ่มเดียวทีเดียว · แก้จำนวน/ราคาได้แม้จองแล้ว (reuse add/cancel charge item).
  • ต้องแก้เยอะไหม → L: ไม่ใช่เพราะซับซ้อน แต่เพราะ Path B (rename ตาราง physical จริง) กระทบกว้าง (~32 ไฟล์ + feature value-added/catalog ที่ ship แล้ว) → retest กว้าง. billing engine ยัง reuse หมด. ดู "Rename Impact & Retest Scope".

สถาปัตยกรรม (architecture)

Terms = อธิบายเงื่อนไข · Property Charge Items = นิยามเงิน · Billing Engine = คิดเงินจริง

Terms / Policy UI→ Property Charge Itemsนิยามเงิน→ Booking Line Items→ Charge / Snapshot / Recompute→ Payment / Invoice / Summary
ศัพท์ที่ใช้ในเล่มนี้ (ฉบับย่อ)
Package (แพ็กเกจ)"สินค้าห้องพัก" ที่ตั้งราคาค่าห้อง + กติกา (จำนวนคนสูงสุด, รับสัตว์ไหม) — ตาราง property_packages
Property charge item (รายการค่าธรรมเนียมของบ้าน)คอนเซ็ปต์กลางของ EPIC นี้ — "เงื่อนไขที่มีค่าธรรมเนียม" 1 อย่าง (สัตว์/แขกเสริม/เด็ก/ผู้มาเยือน/ทำความสะอาด/ล้างจาน). เก็บบน 3 ตาราง (rename จากระบบ value-added): catalog charge_item_master + ราคารายบ้าน property_charge_items + snapshot การจอง booking_charge_items
category (behavior contract)ของใหม่: ชนิดของ charge item ที่กำหนด behavior ชัด — optional_service (สมัครใจ), condition_fee (จากเงื่อนไข เช่น สัตว์), occupancy_fee (จากจำนวนคน เช่น แขกเสริม/เด็ก), mandatory_fee (บังคับ เช่น ทำความสะอาด), rule (ไม่มีเงิน → อยู่ terms). ดูตาราง behavior ในหัวข้อถัดไป
is_system (code ของระบบ)ของใหม่: flag บน catalog บอกว่า code นี้ "ระบบเป็นเจ้าของ" — ลบ/เปลี่ยน code ไม่ได้. seed พื้นฐาน: EXTRA_GUEST, CHILD, VISITOR, PET_SMALL, PET_LARGE, CLEANING, DISHWASHING — เพิ่มได้ไม่จำกัด (เพิ่ม code = เพิ่มชนิด fee)
property_terms (policy layer)ชั้น "อธิบาย/นโยบาย/กฎ" (อนุญาตสัตว์/เด็ก/ผู้มาเยือนไหม, ห้ามสูบบุหรี่, quiet hours). ไม่เก็บเงิน — ถ้ากฎมีค่าใช้จ่าย terms อ้างอิงไปยัง charge item (related_charge_item_id); fee_amount เดิมไม่ใช้เป็นค่าคิดเงิน. เงินมีแหล่งเดียว = charge item
pets_allowed / capacityสวิตช์ "รับสัตว์" + ความจุ (max_extra_guests) — มีอยู่แล้วบน package เป็น gate; ย้ายออกแค่ "ราคา"
Charge (บิลย่อย)1 รายการเงินที่ต้องเก็บ ตาราง booking_charges เช่น deposit, room_balance, fee (ค่าธรรมเนียมเงื่อนไข), service
charge type feeของใหม่: ประเภทบิลของค่าธรรมเนียมเงื่อนไข (occupancy/condition_fee) — แยกจาก service (สมัครใจ) เพื่อให้การเงินแยกรายได้ fee ได้; ชนิดจริงดูจาก charge_item_code บน booking_charge_items
ยอดเรียกเก็บคงเหลือ (balance due)เงินหลังหักมัดจำ = ค่าห้องคงเหลือ + fee เงื่อนไขทั้งหมด · ในระบบ = booking_financial_summary.outstanding_amount (มีอยู่แล้ว)
payment_statusสถานะการจ่ายระดับการจอง เช่น balance_due (ยังมียอดค้าง) — ช่องใหม่บน bookings
pay-all (ชำระพร้อมกันทั้งหมด)จ่าย 1 ครั้งตัดหลายบิลพร้อมกัน — ใช้ payment_allocations (1 payment → หลาย charges, มีอยู่แล้ว)
