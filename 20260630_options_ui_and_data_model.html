Deposit Split (Option B) · สรุป + หน้าจอ + ข้อมูล + schema

มัดจำ 60% + ค่าสัตว์/แขกเสริมเป็น Fee — สรุปครบในที่เดียว

เอกสารฉบับเดียวจบ — ตัดสินใจแล้วว่าทำทางเลือก B (จัดการ "ยอดเรียกเก็บคงเหลือ" เต็มรูป) พร้อม ยก Extra Guest / Pet ออกมาเป็น "Fee" แยกจาก package. รวมทั้ง สรุปผู้บริหาร, หน้าจอ (As-Is เทียบ B), โครงสร้างข้อมูล + database schema (ตาราง+ตัวอย่าง), ประเมินว่าต้องแก้เยอะแค่ไหน (Code Assessment), แผนทำเป็นเฟส, ความเสี่ยง และสิ่งที่ต้องตัดสินใจ. (ทางเลือก A/C ตัดทิ้งแล้ว · ยอดคงเหลือ = บิลที่ยังไม่จ่าย แบบเดิม · ไม่มีระบบแจ้งเตือนลูกค้า)

ค่าห้อง ฿17,000 + แขกเสริม ฿1,500 (2×750) + สัตว์เลี้ยง ฿800 (1 ตัวใหญ่) มัดจำ 60% × ค่าห้อง = ฿10,200 งานโดยรวม: ใหญ่ (L)
สรุป (อ่านก่อน) ศัพท์ (ฉบับย่อ) Fee แยกออกจาก package วันนี้ (As-Is) แบบใหม่ (Option B) Database schema ต้องแก้เยอะไหม แผนทำเป็นเฟส ความเสี่ยง + วิธีคุม เทียบ As-Is vs B สิ่งที่ต้องตัดสินใจ
฿11,580วันนี้: มัดจำตอนจอง60% ของ "ทุกอย่าง" (รวมสัตว์/แขก)
฿10,200แบบใหม่: มัดจำตอนจอง60% ของ "ค่าห้อง" เท่านั้น (−1,380)
฿9,100ยอดเรียกเก็บคงเหลือ (จ่ายทีเดียว)ค่าห้องคงเหลือ + แขกเสริม + สัตว์

อ่านสั้น ๆ ก่อน

  • ตัดสินใจแล้ว: ทำทางเลือก B (จัดการ "ยอดเรียกเก็บคงเหลือ" เต็มรูป — มีสถานะ, ตั้งวันครบกำหนด, ให้ลูกค้าจ่ายล่วงหน้าผ่านลิงก์ได้) — ตัด A/C ทิ้ง.
  • การเปลี่ยนโครงสร้างใหญ่ขึ้น: ยก ค่าแขกเสริม + ค่าสัตว์เลี้ยง ออกมาเป็น "Fee" แยก — เดิมราคาพวกนี้ฝังอยู่ใน package (เป็นช่องในตาราง package) จะย้ายไปอยู่ตารางใหม่ property_fees.
  • package ยังเก็บปุ่มเปิด/ปิด "รับสัตว์เลี้ยง" ไว้เหมือนเดิม — ช่องนี้ (pets_allowed) มีอยู่แล้ว ไม่ต้องทำใหม่; ที่ย้ายออกคือ "ราคา" เท่านั้น.
  • มัดจำยังคิดจาก package (ค่าห้อง) เหมือนเดิม · ค่าแขกเสริม/สัตว์คิดจาก Fee (ตามจำนวนจริง) · admin แก้จำนวน/ค่า fee ได้แม้จองแล้ว.
  • ยอดเรียกเก็บคงเหลือ = ค่าห้องคงเหลือ + แขกเสริม + สัตว์ รวมกัน มีปุ่ม "ชำระพร้อมกันทั้งหมด" ทีเดียว.
  • ต้องแก้เยอะไหม → ใหญ่ (L): ไม่ใช่แค่แก้สูตรมัดจำจุดเดียวแล้ว — เพราะการย้าย fee ออกจาก package = ตารางใหม่ + ย้ายข้อมูล + แก้เครื่องคิดราคา + แก้ 2 หน้าจอ admin; บวกของใหม่ของ B (สถานะ/ลิงก์/ปุ่มจ่ายรวม). ดูหัวข้อ "ต้องแก้เยอะไหม".

แก้บิลหลังจองขัดกฎ "เพิ่มอย่างเดียว ห้ามแก้ของเก่า"สูงคิดใหม่ = ลงรายการปรับปรุง (void + adjustment) ไม่ลบ/แก้ของเดิม; ทดสอบเงินลงตัวหลังแก้
สถานะ "รอเก็บคงเหลือ" ไม่ครอบทุกกรณี (ยกเลิก/หมดอายุระหว่างนั้น)กลางนิยามการเปลี่ยนสถานะให้ชัดในขั้นออกแบบ; ต่อยอดจากวงจรสถานะเดิม

เลขมัดจำหน้าจอ vs หลังบ้านไม่ตรงกลางหน้าจอใช้สูตรเดียวกับหลังบ้าน (60% ของค่าห้อง) + ทดสอบเทียบ 2 ฝั่ง
ตาข่ายความปลอดภัยรวม: เปิดด้วยสวิตช์ (feature flag) ทีละลูกค้าองค์กร เริ่ม 1 ราย → เฝ้าดูเงินลงตัว → ค่อยขยายเต็ม; การจองเก่าแช่แข็ง; มีปัญหาปิดสวิตช์ย้อนได้ (ช่องราคาเก่ายังอยู่ระหว่าง 2 จังหวะ).
